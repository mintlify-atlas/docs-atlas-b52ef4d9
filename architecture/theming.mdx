allocazione.dev uses a cohesive visual system built on a dark bento-grid layout, glassmorphism card styling, Tailwind CSS 4, and a server-injected dynamic accent color.

## Bento-grid layout

The homepage arranges content in a **bento-grid** — a mosaic of cards with varying sizes, inspired by Japanese bento boxes. Each card is a self-contained widget (clock, music player, project links, social links) laid out using CSS Grid.

The grid is fully responsive: columns collapse on smaller viewports while cards retain their individual visual identity.

```svelte
<div class="grid grid-cols-1 md:grid-cols-3 lg:grid-cols-4 gap-4">
  <div class="col-span-2 row-span-2"><!-- large featured card --></div>
  <div class="col-span-1"><!-- small widget --></div>
  <!-- ... -->
</div>
```

## Glassmorphism cards

Every card in the grid uses a consistent glassmorphism style: a semi-transparent dark background, backdrop blur, and a subtle border.

The canonical card class string is:

```html
<div class="bg-[#0f0f0f]/80 backdrop-blur-xl border border-[#252525] rounded-4xl">
  <!-- card content -->
</div>
```

| Class | Effect |
|-------|--------|
| `bg-[#0f0f0f]/80` | Near-black background at 80% opacity |
| `backdrop-blur-xl` | Blurs content behind the card (glassmorphism effect) |
| `border border-[#252525]` | Subtle dark border to define card edges |
| `rounded-4xl` | Heavily rounded corners for a soft, modern look |

<Tip>
  The `backdrop-blur-xl` effect requires the element behind the card to have some visual content. On the solid `#0f0f0f` base background, consider adding a subtle noise texture or gradient to make the blur visible.
</Tip>

## Tailwind CSS 4

The project uses **Tailwind CSS 4**, which shifts all configuration from a JavaScript config file (`tailwind.config.js`) to a CSS-native approach.

Instead of `tailwind.config.js`, design tokens are defined directly in `src/app.css`:

```css
/* src/app.css */
@import "tailwindcss";

@theme {
  --color-background: #0f0f0f;
  --radius-4xl: 2rem;
  /* custom tokens defined here */
}
```

The `@import "tailwindcss"` directive replaces the old `@tailwind base; @tailwind components; @tailwind utilities;` triple import. Vite processes this via the `@tailwindcss/vite` plugin declared in `vite.config.js`.

<Note>
  Tailwind CSS 4 uses CSS `@theme` blocks for custom tokens rather than a `theme.extend` object. If you are familiar with v3, this is the primary migration difference.
</Note>

## Accent color system

The site supports a user-selectable accent color, stored in a cookie and injected at the HTML level before the page loads — preventing a flash of the wrong color.

### How it works

<Steps>
  <Step title="Cookie read in hooks.server.js">
    On every request, `hooks.server.js` reads the `accent_color` cookie. If absent, it falls back to `#e16666` (a muted red).

    ```javascript
    const accentColor = event.cookies.get('accent_color') || '#e16666';
    ```
  </Step>

  <Step title="Hex converted to RGB components">
    The hex value is converted to separate R, G, B integers to support `rgba()` usage in CSS:

    ```javascript
    const hexToRgbStr = (hex) => {
      const r = parseInt(hex.slice(1, 3), 16);
      const g = parseInt(hex.slice(3, 5), 16);
      const b = parseInt(hex.slice(5, 7), 16);
      return `${r}, ${g}, ${b}`;
    };
    ```
  </Step>

  <Step title="Placeholders replaced in HTML">
    The `transformPageChunk` hook replaces template placeholders in `app.html` with the resolved values:

    ```javascript
    transformPageChunk: ({ html }) =>
      html
        .replace('%accent%', accentColor)
        .replace('%accent_rgb%', accentRgb)
    ```

    In `app.html`, these placeholders set CSS custom properties on the `:root`:

    ```html
    <!-- src/app.html -->
    <style>
      :root {
        --accent: %accent%;
        --accent-rgb: %accent_rgb%;
      }
    </style>
    ```
  </Step>

  <Step title="CSS variables used throughout the site">
    Components reference `--accent` and `--accent-rgb` wherever the accent color is needed:

    ```css
    .active-link {
      color: var(--accent);
    }

    .glow-effect {
      box-shadow: 0 0 20px rgba(var(--accent-rgb), 0.3);
    }
    ```
  </Step>

  <Step title="State synced on the client">
    After hydration, the `themeState` store in `$lib/state.svelte.js` is updated to mirror the server-injected values, allowing reactive components to read the accent color:

    ```javascript
    // src/lib/state.svelte.js
    export const themeState = $state({
      accentColor: null,
      accentRgb: null
    });
    ```

    A component can update the accent color (and persist it as a cookie) by writing to `themeState.accentColor`.
  </Step>
</Steps>

## Page entry animation

`app.css` defines an `animate-page-in` utility class that plays a fade-and-translate animation when a page first renders:

```css
@keyframes page-in {
  from {
    opacity: 0;
    transform: translateY(8px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

.animate-page-in {
  animation: page-in 0.35s ease-out both;
}
```

Apply it to the outermost wrapper of a page component:

```svelte
<div class="animate-page-in">
  <!-- page content -->
</div>
```

<Tip>
  Stagger child elements using `animation-delay` to create a cascade effect. anime.js (included in `dependencies`) is available for more complex sequenced animations.
</Tip>

## Security headers

As a side effect of the `hooks.server.js` hook, every response includes hardened security headers:

| Header | Value |
|--------|-------|
| `X-Frame-Options` | `SAMEORIGIN` |
| `X-Content-Type-Options` | `nosniff` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | Camera, microphone, and geolocation disabled |
| `Content-Security-Policy` | Configured per-route allowlist |

These headers are unrelated to visual theming but are set in the same server hook for convenience.
