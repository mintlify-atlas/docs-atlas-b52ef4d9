The site ships with full translations in four languages powered by [`svelte-i18n`](https://github.com/kaisermann/svelte-i18n). Locale detection reads from a cookie first, then falls back to the browser's navigator setting, then to English.

## Supported locales

| Code | Language |
|------|----------|
| `en` | English (default) |
| `it` | Italian |
| `de` | German |
| `es` | Spanish |

## How locale detection works

<Steps>
  <Step title="Read the locale cookie">
    On the client, `getCookie('locale')` reads a `locale` cookie previously set by the site. If the cookie is present and valid, that value is used.
  </Step>

  <Step title="Fall back to browser navigator">
    If no cookie is found, `getLocaleFromNavigator()` returns the browser's preferred language (e.g., `'it-IT'`). The `normalizeLocale` function strips the region subtag and lowercases the result.

    ```js
    const normalizeLocale = (value) => {
      if (!value || typeof value !== 'string') return 'en';
      const normalized = value.toLowerCase().split('-')[0];
      return supportedLocales.has(normalized) ? normalized : 'en';
    };
    ```

    So `'it-IT'` becomes `'it'`, and `'zh-CN'` (unsupported) becomes `'en'`.
  </Step>

  <Step title="Fall back to English">
    If the normalized value is not in the supported set, `normalizeLocale` returns `'en'`. The `init()` call also sets `fallbackLocale: 'en'` so missing keys always resolve to English.
  </Step>
</Steps>

## Hydration and SSR safety

SvelteKit prerenders pages on the server where `window` and `document` are unavailable. To avoid a hydration mismatch, the library always initialises with `initialLocale: 'en'` during SSR.

On the client, after the first animation frame, the saved locale is applied:

```js
const initialLocale = 'en'; // used during SSR

init({
  fallbackLocale: 'en',
  initialLocale,
});

if (typeof window !== 'undefined') {
  if (desiredClientLocale !== initialLocale) {
    requestAnimationFrame(() => {
      locale.set(desiredClientLocale);
    });
  }
}
```

Using `requestAnimationFrame` defers the locale switch until after hydration completes, preventing Svelte from emitting a hydration mismatch warning.

<Note>
  This means Italian visitors will see a brief flash of English on first load before the locale switches. This is a deliberate trade-off to keep SSR output consistent.
</Note>

## Locale persistence

Whenever the active locale changes, a `locale` cookie is written with a one-year expiry:

```js
locale.subscribe((newLocale) => {
  if (newLocale) {
    document.cookie = `locale=${newLocale}; max-age=31536000; path=/; samesite=lax`;
  }
});
```

`max-age=31536000` is 365 days in seconds. On the next visit the cookie is read first (step 1 above) so the user's preferred language is restored immediately.

## Using translations in a component

Import the i18n initialiser and the `t` store at the top of your component. The import has side effects — it registers all message catalogues and calls `init()` — so it only needs to run once.

```svelte
<script>
  import "$lib/locales/i18n.js";
  import { t } from "svelte-i18n";
</script>

<h1>{$t("home.greeting", { values: { name: "Sel" } })}</h1>
<p>{$t("home.who_am_i")}</p>
```

`$t` is a Svelte store. Accessing it with `$` makes the component automatically re-render when the locale changes.

### Interpolation

Pass a `values` object to substitute named placeholders in the message string:

```js
// en.json
{ "home": { "greeting": "hi, i'm {name}." } }
```

```svelte
{$t("home.greeting", { values: { name: "Sel" } })}
// → "hi, i'm Sel."
```

## Translation file structure

Message catalogues live in `src/lib/locales/data/`. Each file is a flat-ish JSON object keyed by page and then by string identifier:

```json
{
  "nav": {
    "home": "home",
    "projects": "projects",
    "uses": "/uses",
    "contact": "contact"
  },
  "home": {
    "title": "@seleneftw / home",
    "greeting": "hi, i'm {name}.",
    "short_for": "short for Selene",
    "who_am_i": "who am i?"
  },
  "projects": {
    "title": "...",
    "header": "the archive."
  },
  "contact": {
    "title": "...",
    "header": "contact."
  },
  "uses": {
    "title": "...",
    "header": "/uses.",
    "categories": []
  }
}
```

All four catalogues (`en.json`, `it.json`, `de.json`, `es.json`) follow the same shape. Missing keys fall back to the English catalogue.

## How messages are registered

All four catalogues are registered in `src/lib/locales/i18n.js` using `addMessages`:

```js
import { addMessages, init, getLocaleFromNavigator, locale } from 'svelte-i18n';
import enMessages from './data/en.json';
import itMessages from './data/it.json';
import deMessages from './data/de.json';
import esMessages from './data/es.json';

addMessages('en', enMessages);
addMessages('it', itMessages);
addMessages('de', deMessages);
addMessages('es', esMessages);
```

## Adding a new locale

<Steps>
  <Step title="Create the translation file">
    Add a new JSON file under `src/lib/locales/data/`, for example `fr.json`. Copy `en.json` as a starting point and translate every value.

    ```
    src/lib/locales/data/
    ├── en.json
    ├── it.json
    ├── de.json
    ├── es.json
    └── fr.json  ← new
    ```
  </Step>

  <Step title="Register the messages">
    Open `src/lib/locales/i18n.js` and add an import and an `addMessages` call:

    ```js
    import frMessages from './data/fr.json';
    addMessages('fr', frMessages);
    ```
  </Step>

  <Step title="Add to the supported set">
    Add the new locale code to `supportedLocales` so that `normalizeLocale` accepts it:

    ```js
    const supportedLocales = new Set(['en', 'it', 'de', 'es', 'fr']);
    ```
  </Step>

  <Step title="Expose a locale switcher (optional)">
    Import `locale` from `svelte-i18n` and call `locale.set('fr')` from any UI control. The cookie subscriber will persist the selection automatically.

    ```svelte
    <script>
      import { locale } from 'svelte-i18n';
    </script>

    <button onclick={() => locale.set('fr')}>Français</button>
    ```
  </Step>
</Steps>

<Tip>
  Keep translation keys in lower_snake_case and group them by page or feature. Avoid deeply nested objects — two levels (`page.key`) is sufficient for most cases.
</Tip>
